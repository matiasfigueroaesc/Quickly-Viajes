1. QUE SE HIZO
-------------------------------------------------------------------------

Se reemplazó completamente H2 (base de datos en memoria/archivo local) por 
MySQL como motor de datos de producción. El cambio es transparente y mantiene 
compatibilidad con Flyway para migraciones automáticas.

a) Dependencias actualizadas en pom.xml (11 microservicios)
   
   Removidas:
   - com.h2database:h2 (mantenido en runtime para perfil h2)
   
   Agregadas:
   - com.mysql:mysql-connector-j 8.0.33 (driver JDBC para MySQL)
   
   Microservicios afectados:
     * ms-pasajeros, ms-conductores, ms-viajes, ms-pagos, ms-tarifas
     * ms-calificaciones, ms-incidencias, ms-vehiculos, ms-zonas
     * ms-notificaciones, ms-auth

b) Configuración MySQL en application.yml (11 microservicios)
   
   Nueva configuración principal (por defecto):
   
     spring:
       datasource:
         url: jdbc:mysql://localhost:3306/{nombre_db}?useSSL=false&serverTimezone=UTC&allowPublicKeyRetrieval=true
         driverClassName: com.mysql.cj.jdbc.Driver
         username: root
         password: root
       jpa:
         database-platform: org.hibernate.dialect.MySQL8Dialect
         hibernate:
           ddl-auto: validate
         show-sql: true
       flyway:
         enabled: true
         baseline-on-migrate: true
   
   Detalles de configuración:
   - useSSL=false: desactivado para desarrollo local (ACTIVAR en producción)
   - serverTimezone=UTC: evita errores de timezone en MySQL 5.7+
   - allowPublicKeyRetrieval=true: requerido en MySQL 8.0+ con creds estándar
   - database-platform MySQL8Dialect: optimizado para MySQL 8.0
   - ddl-auto: validate (no crear tablas, usar Flyway)
   
   Perfil H2 (para desarrollo sin MySQL):
   - Incluido en application.yml (spring.config.activate.on-profile: h2)
   - Se activa con: -Dspring.profiles.active=h2
   - Mantiene configuración H2 original (archivo local en ./data/{service}db)

d) Migraciones Flyway actualizadas (11 archivos V1__*.sql)
   
   Cambios en cada V1__crear_tabla_*.sql:
   
   Antes (compatible H2):
   ```
   CREATE TABLE pasajeros (
       id BIGINT AUTO_INCREMENT PRIMARY KEY,
       nombre VARCHAR(100) NOT NULL,
       ...
   );
   ```
   
   Después (optimizado para MySQL):
   ```
   CREATE TABLE pasajeros (
       id BIGINT AUTO_INCREMENT PRIMARY KEY,
       nombre VARCHAR(100) NOT NULL,
       email VARCHAR(100) NOT NULL UNIQUE,
       telefono VARCHAR(20) NOT NULL,
       created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
       updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
       INDEX idx_email (email),
       INDEX idx_nombre (nombre)
   ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
   ```
   
   Mejoras implementadas:
   - Engine=InnoDB: motor transaccional estándar en MySQL
   - Charset=utf8mb4: soporte completo de caracteres Unicode
   - Collation=utf8mb4_unicode_ci: comparación case-insensitive Unicode
   - created_at / updated_at: timestamps para auditoría
   - INDEX: índices en columnas frecuentemente consultadas
   - CHECK constraints: validación de datos (ej: puntuación 1-5)
   - UNIQUE constraints: unicidad de campos
   
   Tablas actualizadas:
   - pasajeros (id, nombre, email, telefono, timestamps, índices)
   - conductores (id, nombre, email, telefono, licencia, activo, índices)
   - viajes (id, pasajero_id, conductor_id, origen, destino, estado, monto, índices)
   - pagos (id, viaje_id, monto, metodo, estado, índices)
   - tarifas (id, zona, precios, activa, índices)
   - calificaciones (id, viaje_id, pasajero_id, conductor_id, puntuacion, comentario)
   - incidencias (id, viaje_id, tipo, descripcion, estado, índices)
   - vehiculos (id, conductor_id, patente, marca, modelo, anio, tipo, activo, índices)

2. DECISIONES TECNICAS TOMADAS
-------------------------------------------------------------------------

   a) MySQL 8.0 como motor de producción
      - Compatible con Laragon (última versión de MySQL)
      - Driver: mysql-connector-j 8.0.33 (recomendado oficial)
      - Alternativa a Oracle, PostgreSQL, H2
   
   b) Charset UTF8MB4 en lugar de UTF8
      - UTF8 en MySQL es limitado (3 bytes, solo BMP Unicode)
      - UTF8MB4 es el estándar recomendado (4 bytes, todo Unicode + emojis)
      - Collation: utf8mb4_unicode_ci (case-insensitive, locale-aware)
   
   c) Engine InnoDB en todas las tablas
      - Transacciones ACID (importante para pagos y viajes)
      - Foreign keys (si implementamos en FASE 5+)
      - Row-level locking (mejor concurrencia)
   
   d) Timestamps automáticos (created_at, updated_at)
      - created_at: TIMESTAMP DEFAULT CURRENT_TIMESTAMP (inmutable)
      - updated_at: TIMESTAMP ON UPDATE CURRENT_TIMESTAMP (actualiza con cada cambio)
      - Útil para auditoría y debugging
   
   e) Índices en columnas de consulta frecuente
      - email, username: búsquedas por identificador
      - estado, activo: filtros comunes
      - conductor_id, pasajero_id: joins
      - Reduce tiempo de query, aumenta performance
   
   f) Perfil H2 opcional
      - Permite desarrollo local sin MySQL instalado
      - Activado con: -Dspring.profiles.active=h2
      - application.yml incluye dos configuraciones (default: MySQL, h2: H2)
      - Facilita CI/CD y testing sin dependencias externas
   
   g) Datos de prueba en Flyway
      - INSERTs incluidos en V1__*.sql
      - Permite testing inmediato sin seeders separados
      - Datos realistas pero ficticios (Juan Perez, etc.)
   
   h) Configuración Flyway sin cambios
      - enabled: true (automático)
      - baseline-on-migrate: true (solo primera vez)
      - Versionado con V1__*, V2__*, etc. (estándar Flyway)

5. ESTRUCTURA DE DATOS FINAL
-------------------------------------------------------------------------

Tabla: pasajeros (11 registros ejemplo: 1 seed + nuevos)
- id: BIGINT (PK, Auto)
- nombre: VARCHAR(100)
- email: VARCHAR(100) UNIQUE
- telefono: VARCHAR(20)
- created_at: TIMESTAMP (auto)
- updated_at: TIMESTAMP (auto)
- Índices: idx_email, idx_nombre

Tabla: conductores (1 seed)
- id: BIGINT
- nombre, email, telefono, licencia (UNIQUE)
- activo: BOOLEAN
- Índices: idx_email, idx_licencia, idx_activo

Tabla: viajes (1 seed)
- id, pasajero_id, conductor_id, origen, destino
- estado: ENUM-like VARCHAR(30)
- monto_estimado: DECIMAL(10,2)
- Índices: idx_pasajero, idx_conductor, idx_estado

Tabla: pagos (1 seed)
- id, viaje_id, monto: DECIMAL(10,2)
- metodo, estado: VARCHAR
- Índices: idx_viaje, idx_estado

Tabla: tarifas (1 seed)
- id, zona (UNIQUE), precio_base, precio_por_km, precio_por_minuto: DECIMAL
- activa: BOOLEAN

Tabla: calificaciones (1 seed)
- id, viaje_id, pasajero_id, conductor_id
- puntuacion: INT (CHECK 1-5)
- comentario: TEXT (nullable)

Tabla: incidencias
- id, viaje_id, tipo, descripcion, estado
- Índices: idx_viaje, idx_tipo, idx_estado

Tabla: vehiculos (1 seed)
- id, conductor_id, patente (UNIQUE), marca, modelo, anio
- tipo, activo
- Índices: idx_conductor, idx_patente, idx_activo

Tabla: zonas (1 seed)
- id, nombre (UNIQUE), descripcion, latitud, longitud
- activa: BOOLEAN

Tabla: notificaciones
- id, usuario_id, tipo, mensaje, leida
- Índices: idx_usuario, idx_leida, idx_tipo

Tabla: usuarios (1 seed: admin)
- id, username (UNIQUE), email (UNIQUE), password (hash BCrypt 255 chars)
- rol, activo: BOOLEAN
- Índices: idx_username, idx_email, idx_activo

Todas las tablas:
- Engine: InnoDB
- Charset: utf8mb4
- Collation: utf8mb4_unicode_ci
- Timestamps: created_at, updated_at (excepto usuarios que ya tenía)

6. PERFORMANCE Y OPTIMIZACIONES
-------------------------------------------------------------------------

Índices creados:
- Búsquedas por email: O(log n) en lugar de O(n)
- Búsquedas por estado/activo: O(log n)
- Joins por FK: O(log n)

Estimaciones de query:
- GET /api/pasajeros: ~5ms (con índice en email)
- GET /api/pasajeros/{id}: ~1ms (PK)
- POST /api/pasajeros: ~10ms (insert + indexes)
- SELECT COUNT: ~1ms (puede optimizar con redis en FASE 6)

Posibles mejoras futuras:
- Redis para caché de tarifas
- Particionamiento de tabla viajes por fecha
- Read replicas para queries reportes
- Slow query log para identificar cuellos de botella
