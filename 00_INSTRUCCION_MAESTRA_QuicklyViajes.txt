1. METODOLOGIA DE TRABAJO (LEER PRIMERO)
===============================================================================

El desarrollo se hace EN FASES, paso a paso, NUNCA todo de una vez:

1. Antes de programar una fase, se presenta un plan breve y se espera el
   visto bueno explicito del usuario.
2. Si una tarea requiere accion del propio usuario (instalar algo, crear una
   cuenta, ejecutar un comando en su maquina, configurar Laragon, etc.), se le
   dan instrucciones claras y numeradas en vez de asumir que la IA puede
   hacerlo.
3. Al terminar cada fase, se genera un archivo .txt de resumen (mismo estilo
   que este documento) que describe: que se hizo, que decisiones se tomaron,
   que quedo pendiente, y como retomar. Estos resumenes se numeran
   correlativamente, ej:
       01_FASE0_resumen.txt
       02_FASE1_resumen.txt
       03_FASE2_resumen.txt
       etc.
3.1. IMPORTANTE - Entrega de resultados junto con el resumen:
   El archivo .txt de resumen NUNCA se entrega solo. Cada vez que se cierra
   una fase, se debe entregar EN EL MISMO MOMENTO, sin que el usuario tenga
   que pedirlo:
     a) El archivo .txt de resumen de la fase.
     b) TODOS los archivos/codigo resultantes de esa fase (codigo fuente
        nuevo o modificado, pom.xml, application.yml, migraciones, etc.),
        ya sea como archivos individuales o empaquetados en un .zip del
        proyecto completo actualizado, segun lo que tenga mas sentido para
        el volumen de cambios.
     c) Si se hicieron cambios sobre el repositorio git (commits), el
        commit debe quedar incluido dentro del .zip entregado (no basta con
        mencionar el hash del commit en el texto).
   Esta regla existe porque en fases anteriores el resumen .txt se entrego
   sin los archivos/resultados reales, obligando al usuario a pedirlos por
   separado. Eso no debe volver a ocurrir: el resumen y los resultados se
   entregan siempre juntos, en la misma respuesta de cierre de fase.
4. No se avanza a la fase siguiente sin que el usuario lo confirme.
5. Cualquier persona (o IA) que retome el proyecto debe leer primero este
   archivo y luego el ULTIMO resumen de fase generado, para saber exactamente
   en que punto esta el desarrollo.

3. DECISIONES TECNICAS YA ACORDADAS CON EL USUARIO
===============================================================================

a) Base de datos de produccion: MySQL, usando Laragon en el entorno local
   de desarrollo del usuario (servidor MySQL de Laragon, puerto 3306 por
   defecto). En Docker/Render se usara un contenedor MySQL equivalente,
   manteniendo compatibilidad de conexion (mismo dialecto, mismas
   migraciones Flyway).

b) Estrategia de despliegue: primero LOCAL con docker-compose (probar que
   todo el sistema levanta sin IDE), y despues se conecta GitHub con Render
   para despliegue continuo en la nube.

c) Autenticacion: se crea un microservicio nuevo "ms-auth" que:
   - Gestiona usuarios/credenciales con Spring Security.
   - Hashea contraseñas (BCrypt).
   - Emite tokens JWT al autenticar.
   El API Gateway es responsable de validar el JWT en cada request entrante
   ANTES de enrutar hacia el microservicio correspondiente. Los
   microservicios internos no vuelven a validar el JWT (confian en que el
   Gateway ya filtro las peticiones). Este es el enfoque "Gateway como
   punto unico de validacion", mas simple y adecuado para el alcance del
   proyecto.
